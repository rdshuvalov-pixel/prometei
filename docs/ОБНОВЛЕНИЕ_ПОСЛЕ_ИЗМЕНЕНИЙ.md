# Обновление после изменений в репозитории

Краткий порядок: **git → Vercel (веб) → VPS (воркер)**. Делай только то, что ты менял.

---

## 1. Локально и Git

1. Проверка: `git status`, при необходимости `git diff`.
2. Коммит в ветку, с которой работаешь (например `main`).
3. **Push** в удалённый репозиторий: `git push origin <ветка>`.

---

## 2. Веб (Vercel)

Если менялся **`apps/web`** (страницы, API, `package.json`):

1. После push Vercel обычно **сам запускает деплой**, если проект привязан к репо.
2. Дождись успешного **Deployments** в панели Vercel; при сбое — **Redeploy** (при смене env — без кэша).
3. Если добавлял переменные окружения — **Settings → Environment Variables → Production**, затем снова **Redeploy**.

Корень проекта на Vercel должен быть **`apps/web`** (см. [`README.md`](../README.md)).

---

## 3. Воркер на VPS (Docker)

Если менялось **`prometheus_agent/`**, **`worker/`**, **`Dockerfile.worker`**, **`docker-compose.worker.yml`**, **`requirements-worker.txt`**:

1. **SSH** на сервер, перейди в каталог с клоном репозитория (тот же путь, где лежит `docker-compose.worker.yml`).
2. Подтянуть код: `git pull` (та же ветка, что и push).
3. Пересобрать и поднять контейнер:
   ```bash
   docker compose -f docker-compose.worker.yml up -d --build
   ```
4. Убедиться, что сервис жив:  
   `docker compose -f docker-compose.worker.yml ps`  
   Логи:  
   `docker compose -f docker-compose.worker.yml logs --tail=100 worker`

**Перезапуск без пересборки** (например только сменил `.env.worker` на хосте):

```bash
docker compose -f docker-compose.worker.yml up -d
```

или жёстко:

```bash
docker compose -f docker-compose.worker.yml restart worker
```

После **`--build`** образ обновится с новым кодом; **`restart`** подхватит тот же образ и новый **env** из файла.

---

## 4. Только markdown-цели (без пересборки образа)

Если правил только **`prometheus_agent/search_targets.md`** / **`watchlist_targets.md`** и они **смонтированы в контейнер** (volume на весь `./prometheus_agent` или ты копируешь файлы на сервер в примонтированный `out`/родительский каталог — как у тебя настроено):

- При типичной схеме из репо в образ **копируется** `prometheus_agent` при **build** → нужен снова **`docker compose ... up -d --build`**, либо volume на каталог `prometheus_agent` с хоста.

Проверь в **`docker-compose.worker.yml`**: сейчас в репо смонтирован в основном **`./prometheus_agent/out`**. Значит правки **`.md` целей** попадут в контейнер **только после rebuild** образа (или добавь volume на `./prometheus_agent` сам, если хочешь править цели без rebuild).

---

## 5. Cron enqueue (Vercel API)

Если менялся только URL/cron на VPS, но не код воркера — достаточно актуального **`scripts/enqueue-cron.example.sh`** и переменных **`VERCEL_URL`**, **`ENQUEUE_SECRET`** на машине с cron.

---

## 6. Файл ротации crawl

`prometheus_agent/out/crawl_url_cursor.json` создаётся воркером при **`CRAWL_ROTATE_URLS=1`** (по умолчанию) и лимите **`MAX_CRAWL_URLS`**. Лежит в томе **`out`** — не в git. Сбросить очередь URL «с начала» — удали этот файл на сервере (или выставь **`CRAWL_ROTATE_URLS=0`** для старого поведения «всегда первые N»).
