# Обновление после изменений в репозитории

Краткий порядок: **git → Vercel (веб) → VPS (воркер)**. Делай только то, что ты менял.

**Полная пошаговая установка воркера на VPS** (первый раз, SSH, клон, `.env.worker`, проверки): [`ИНСТРУКЦИЯ_VPS_ВОРКЕР_CRON.md`](ИНСТРУКЦИЯ_VPS_ВОРКЕР_CRON.md) — разделы **3–5** и далее.

Ниже — **только цепочка обновления**, когда сервер уже настроен.

---

## 1. Локально и Git

1. Проверка: `git status`, при необходимости `git diff`.
2. Коммит в ветку, с которой работаешь (например `main`).
3. **Push** в удалённый репозиторий: `git push origin <ветка>`.

---

## 2. Веб (Vercel)

Если менялся **`apps/web`** (страницы, API, `package.json`):

1. После push Vercel обычно **сам запускает деплой**, если проект привязан к репо.
2. Дождись успешного **Deployments** в панели Vercel; при сбое — **Redeploy** (при смене env — без кэша).
3. Если добавлял переменные окружения — **Settings → Environment Variables → Production**, затем снова **Redeploy**.

Корень проекта на Vercel должен быть **`apps/web`** (см. [`README.md`](../README.md)).

---

## 3. Воркер на VPS (Docker) — шаг за шагом

Нужно, если менялось **`prometheus_agent/`**, **`worker/`**, **`Dockerfile.worker`**, **`docker-compose.worker.yml`**, **`requirements-worker.txt`** (или хочешь подтянуть уже запушенный коммит на сервер).

Путь в примерах: **`/opt/prometei`** (как в [ИНСТРУКЦИЯ_VPS…](ИНСТРУКЦИЯ_VPS_ВОРКЕР_CRON.md)); у себя подставь **свой** каталог, где лежит `docker-compose.worker.yml`.

| Шаг | Действие | Зачем |
|-----|----------|--------|
| 1 | `ssh user@твой-сервер` | Подключение к VPS. |
| 2 | `cd /opt/prometei` | Оказаться **в корне клона** репозитория (рядом `docker-compose.worker.yml`, `Dockerfile.worker`). |
| 3 | `git status` (по желанию) | Убедиться, что нет лишних локальных правок; иначе `git stash` или разбери вручную. |
| 4 | `git pull origin main` | **Только скачать коммиты** с GitHub; Docker и контейнеры **ещё не** обновляются. |
| 5 | `docker compose -f docker-compose.worker.yml up -d --build` | **`--build`** — пересобрать образ с новым кодом из диска. **`up -d`** — пересоздать/запустить сервис **`worker`** в фоне. Это **одна** команда, её можно разбить (см. ниже). |
| 6 | `docker compose -f docker-compose.worker.yml ps` | Проверить, что контейнер **Up**. |
| 7 | `docker compose -f docker-compose.worker.yml logs --tail=100 worker` | Убедиться, что воркер стартовал и нет явной ошибки (Supabase, путь `WORKER_CMD` и т.д.). |

**Тот же смысл, что шаг 5, но двумя командами** (удобно смотреть, на чём упал build):

```bash
cd /opt/prometei
docker compose -f docker-compose.worker.yml build
docker compose -f docker-compose.worker.yml up -d
```

Или одной строкой, как в шаге 5: `... up -d --build` (= сначала build при необходимости, потом up).

**Только `git pull`, без Docker** — на сервере **останется старый образ**; новый Python/markdown в контейнере не появится, пока не сделаешь шаг 5 (или `build`+`up`).

---

### Когда не нужен `--build`

- Менял **только** `.env.worker` на хосте (и код образа тот же):

```bash
cd /opt/prometei
docker compose -f docker-compose.worker.yml up -d
```

либо

```bash
docker compose -f docker-compose.worker.yml restart worker
```

- **`up -d` / `restart`** перечитает **env** из **`.env.worker`**, примонтированного в `docker-compose`, но **не** подтянет новый код с GitHub: для нового кода всё равно нужен **`git pull`** + **`--build`** (или pull + `build` + `up`).

---

## 4. Только markdown-цели (без пересборки образа)

Если правил только **`prometheus_agent/search_targets.md`** / **`watchlist_targets.md`** и они **смонтированы в контейнер** (volume на весь `./prometheus_agent` или ты копируешь файлы на сервер в примонтированный `out`/родительский каталог — как у тебя настроено):

- При типичной схеме из репо в образ **копируется** `prometheus_agent` при **build** → нужен снова **`docker compose ... up -d --build`**, либо volume на каталог `prometheus_agent` с хоста.

Проверь в **`docker-compose.worker.yml`**: сейчас в репо смонтирован в основном **`./prometheus_agent/out`**. Значит правки **`.md` целей** попадут в контейнер **только после rebuild** образа (или добавь volume на `./prometheus_agent` сам, если хочешь править цели без rebuild).

---

## 5. Cron enqueue (Vercel API)

Если менялся только URL/cron на VPS, но не код воркера — достаточно актуального **`scripts/enqueue-cron.example.sh`** и переменных **`VERCEL_URL`**, **`ENQUEUE_SECRET`** на машине с cron.

---

## 6. Файл ротации crawl

`prometheus_agent/out/crawl_url_cursor.json` создаётся воркером при **`CRAWL_ROTATE_URLS=1`** (по умолчанию) и лимите **`MAX_CRAWL_URLS`**. Лежит в томе **`out`** — не в git. Сбросить очередь URL «с начала» — удали этот файл на сервере (или выставь **`CRAWL_ROTATE_URLS=0`** для старого поведения «всегда первые N»).
